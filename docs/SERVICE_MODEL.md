# Service Model

## Purpose

This document defines the high-level service model for OaaS — Operations as a Service.

The service model describes what is operated, what is included, what is excluded and how operational responsibility is made explicit.

---

## Service scope

OaaS is designed for small regulated IT, platform, DevSecOps or infrastructure-heavy environments where operational clarity matters.

Typical scope:

- service ownership model
- runbook structure
- incident response model
- change governance
- backup and recovery evidence
- service review cadence
- handover readiness
- audit evidence examples

---

## Out of scope

OaaS does not define:

- customer-specific production commitments
- real SLA contracts
- real incident history
- real customer architecture
- secret handling procedures for a live environment
- organization-specific legal commitments

Those belong in a private annex or customer-specific service agreement.

---

## Service boundary

The model separates three layers:

| Layer | Description |
| --- | --- |
| Public reference model | Generic templates, examples and operating principles |
| Private operational annex | Real environment details, contacts, systems, evidence and approvals |
| Customer/service agreement | Formal commercial and contractual commitments |

---

## Core principle

A service is not operationally mature until another qualified person can understand what it does, how it is supported, how incidents are handled and how recovery is verified.
