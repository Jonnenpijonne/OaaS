# SLA and SLO Model

## Purpose

This document defines lightweight SLA and SLO thinking for OaaS.

The goal is not to create a legal contract, but to make expectations visible.

---

## Terms

| Term | Meaning |
| --- | --- |
| SLA | Formal service-level agreement or commitment |
| SLO | Internal service-level objective |
| SLI | Indicator used to measure service behavior |

---

## Example SLO areas

| Area | Example objective |
| --- | --- |
| Availability | Service should be available during agreed operating window |
| Incident response | High-priority issues should be acknowledged within agreed time |
| Recovery | Restore process should be tested and evidenced |
| Change handling | Changes should have owner, scope and rollback plan |
| Documentation | Runbooks should be current enough for handover |

---

## SLO review questions

```text
What does good operation mean for this service?
What can be measured?
What evidence exists?
What is only an internal target?
What is a formal commitment?
What needs customer-specific agreement?
```

---

## Boundary

Public examples should describe objectives, not real customer commitments.

Formal SLA terms belong in customer-specific agreements.
