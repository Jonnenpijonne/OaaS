# Public / Private Boundary

## Purpose

This document defines what belongs in the public OaaS repository and what must stay in a private operational annex.

The goal is to make the repository useful as a portfolio-safe reference without exposing real operational material.

---

## Public repository may contain

- generic service model
- generic operating model
- synthetic examples
- reusable templates
- public-safe evidence examples
- high-level governance principles
- role-based responsibility examples

---

## Private annex should contain

- real service names
- real contact paths
- real environment details
- real incident records
- real recovery evidence
- real operational decisions
- real customer-specific commitments
- internal service review material

---

## Boundary rule

The public repository should explain the model.

The private annex should hold the real operational evidence.

---

## Review question

Before adding material to the public repository, ask:

```text
Could this reveal a real customer, system, incident, location, relationship, weakness or operational dependency?
```

If yes, keep it private or rewrite it as a synthetic example.
